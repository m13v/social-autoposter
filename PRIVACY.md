# Privacy

`social-autoposter` is open source and runs on your own machine. It posts
content on your behalf to the social platforms you authenticate with locally.
This document describes exactly what data leaves your machine, when, and why.

## TL;DR

- The pipeline only "phones home" when it actually calls the
  `social-autoposter-website` API (post log, reply state transitions, DM
  state, engagement-style telemetry).
- When it does, it sends a stable install ID plus a small machine snapshot,
  so the server can attribute and rate-limit your requests without requiring
  you to sign up.
- No browser cookies, OAuth tokens, post drafts, or DM contents are sent
  beyond what already lives in the database you control via `DATABASE_URL`.

## Local-only data

The pipeline keeps most of its state on your disk:

- `~/.social-autoposter/identity.json` (your install fingerprint; see below)
- `~/social-autoposter/.env` (DATABASE_URL, API keys, browser session paths)
- `~/.claude/browser-profiles/*` (logged-in browser sessions for Reddit, X,
  LinkedIn, etc.)

None of those files are uploaded.

## What is sent to social-autoposter-website

Every `/api/v1/*` HTTP call carries an `X-Installation` header containing
base64-encoded JSON of:

| Field | How we collect it | Why we keep it |
|---|---|---|
| `install_id` | random UUID v4 generated by `scripts/identity.py` on first run | attribute writes per install; rate-limit |
| `hardware_uuid` | `ioreg IOPlatformUUID` (macOS), `/etc/machine-id` (linux), `MachineGuid` (windows) | dedupe across reinstalls of the repo |
| `hostname` | `scutil --get ComputerName` or `socket.gethostname()` | support and debugging |
| `os`, `os_version`, `cpu_arch` | `platform.*` | bug attribution per OS/arch |
| `python_version`, `node_version` | local interpreters | bug attribution per runtime |
| `git_email` | `git config --global user.email`, only if set | optional contact for support |
| `tz` | system timezone | scheduling diagnostics |

The server additionally records, at request time:

- IP address (any TCP server sees this; we just persist it)
- `User-Agent` header
- Country / city / region inferred from the IP if the request hits Vercel's
  edge

## What is NOT collected

- Browser cookies or OAuth tokens (those stay in your local browser
  profile)
- The text of your scheduled posts beyond what already goes into your own
  Neon database via `DATABASE_URL`
- Keystrokes, screen captures, or microphone input
- Any analytics ping unless the pipeline is actively calling the API

## Deleting your data

To rotate your local install_id:

```bash
python3 ~/social-autoposter/scripts/identity.py reset
```

That deletes `~/.social-autoposter/identity.json` and forces a fresh
`install_id` on the next call.

To delete the server-side row, open an issue at
https://github.com/m13v/social-autoposter (or email i@m13v.com) with the
install_id printed by `python3 scripts/identity.py show`. Self-serve
deletion via API is on the roadmap.

## Rate limits

Each `install_id` is capped at 5,000 write requests per rolling 24h window.
The cap is shared across all `/api/v1/*` endpoints. Excess requests come
back as `429 rate_limited`.
